Two APIs. Pick Your Style.

A declarative YAML DSL for simple flows, and a dynamic async Python API for scripts that need to branch on what's actually on screen.

Template matching + OCR

Register UI elements by cropping them from screenshots in the web editor. phingr finds them in the live camera feed using OpenCV template matching. Use OCR (text:) for anything with readable labels — no accessibility IDs required.

  • Dynamic Python: async with PhingrSession(...)
  • YAML DSL: tapOn, swipeUntilFound, repeat…
  • Find with bounding box + match score
  • Branch on UI state: if await s.exists("btn")
  • Live MJPEG screen capture stream

import asyncio
from phingr import PhingrSession

SERVER_URL = "http://localhost:8800"
DEVICE_URL = "http://phingr-e825.local:8080"


async def run_once(s: PhingrSession, i: int) -> bool:
    print(f"\n── Iteration {i} ──")

    await s.press_key("home")
    await s.wait(1)

    # Swipe right to the photo widget (proves we're on home screen)
    await s.swipe_until_found("photo_widget", direction="RIGHT", max_swipes=4)
    if not await s.exists("photo_widget"):
        return False

    # Swipe left to Settings, verified by fit_icon context
    await s.swipe_until_found("settings_icon", direction="LEFT", max_swipes=4)
    if not await s.exists("fit_icon"):
        return False

    await s.tap_on("settings_icon")
    await s.swipe_until_found("bt_header", direction="DOWN", max_swipes=2)

    # Look at both template and OCR — find() returns bbox + score
    on = await s.find("bt_toggle_on")
    off = await s.find("bt_toggle_off")
    text_hits = await s.find_text("Bluetooth")
    print(f"on={on}  off={off}  ocr={[t['text'] for t in text_hits]}")

    # Branch on what's actually on screen
    if on:
        await s.tap_on("bt_toggle_on", offset=(0.85, 0.5))
    elif off:
        await s.tap_on("bt_toggle_off", offset=(0.85, 0.5))
    else:
        return False

    await s.wait(3)
    # Second toggle — OR expression, lower threshold (state just changed)
    await s.tap_on("(bt_toggle_on, bt_toggle_off)",
                    offset=(0.85, 0.5), threshold=0.6)
    return True


async def main(iterations: int = 100):
    async with PhingrSession(SERVER_URL, DEVICE_URL, name="bt-toggle-stress") as s:
        passed = failed = 0
        for i in range(1, iterations + 1):
            try:
                if await run_once(s, i):
                    passed += 1
                else:
                    failed += 1
            except Exception as e:
                print(f"iter {i} ERROR: {e}")
                failed += 1
        print(f"Done: {passed} passed, {failed} failed")


asyncio.run(main())

Built on Three Principles

01

Not XCTest. Full Access.

XCTest-based tools run inside Apple's sandbox. They can't touch the lock screen, can't open System Settings, can't toggle Wi-Fi or Bluetooth. phingr uses USB HID — it controls iPhones as a physical mouse and keyboard. If a human finger can reach it, phingr can automate it. Lock screen, system settings, third-party apps, alert dialogs — everything.

02

Edge-Only. Zero Cloud.

Everything runs on a Raspberry Pi sitting next to your mobile device. There is no cloud component. No telemetry. No external API calls. No accounts on someone else's server. Your test data, your screen content, your device interactions - none of it ever leaves your network. This is not a feature. It is the architecture.

03

Open Source. Modify Freely.

The full source code is on GitHub. With an active subscription, you can read it, modify it, extend it, and adapt it to your workflow. Add custom API endpoints. Change the OCR pipeline. Integrate with your internal tools. The code is yours to work with. No black boxes.

What phingr Is (and Isn't)

phingr is

  • An iPhone UI test automation tool
  • USB HID-based — full access, no sandbox
  • Edge-only — runs entirely on a Raspberry Pi
  • YAML DSL + async Python API (PhingrSession)
  • Template matching + OCR for element finding
  • Live MJPEG stream + web UI for flow editing
  • Open source with subscription

phingr is not

  • Based on XCTest or XCUITest
  • Compatible with Android (iOS only — Android's HID stack is too strict)
  • A general-purpose remote desktop
  • A cloud service or SaaS platform
  • Something that requires Xcode or a Mac
  • Something that requires jailbreaking
